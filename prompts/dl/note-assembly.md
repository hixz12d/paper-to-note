# 阶段 4：DL 笔记组装

## 任务

将前三个阶段的产出组装成一份完整的 DL 论文 Markdown 笔记，可直接进入 Obsidian vault。

## 规则

### 1. 输入

你已经在当前对话中产出了以下内容：
1. 阶段 1：元数据 + 论文结构 + 关键原文片段
2. 阶段 2：Figure 论证路径分析
3. 阶段 3：模型架构与训练 + Benchmark 与 Ablation
4. （如为交互模式）用户讨论内容

### 2. 模板

参考 `templates/dl-note-template.md` 的骨架结构，将各阶段产出填入对应区块。

### 3. 语言规则（最高优先级，严格执行）

1. 正文说明用中文撰写
2. 以下内容保留英文：
   - 模型名 / 架构名：`ResNet`, `ViT`, `YOLO`, `U-Net`, `DETR`, `SAM`, `GAN`, `Diffusion`
   - 指标缩写：`mAP`, `IoU`, `FID`, `PSNR`, `SSIM`, `BLEU`, `Top-1`, `AUC`
   - 数据集名：`ImageNet`, `COCO`, `VOC`, `ADE20K`, `Cityscapes`, `CIFAR-10`
   - 层 / 模块名：`Conv`, `BN`, `LN`, `ReLU`, `GELU`, `Softmax`, `MLP`, `Patch Embedding`
   - 优化器名 / 损失函数名在作为专有值时可保留英文，如 `AdamW`, `SGD`, `Cross-Entropy`
3. 以下常见术语统一译成中文：
   - `feature map` → 特征图
   - `backbone` → 骨干网络
   - `attention` → 注意力
   - `pooling` → 池化
   - `skip connection` → 跳跃连接
   - `head` → 检测头 / 任务头（按上下文）
   - `neck` → 特征融合层
   - `anchor` → 锚框
   - `embedding` → 嵌入
   - `fine-tuning` → 微调
   - `pre-training` → 预训练
   - `inference` → 推理
   - `epoch` → 训练轮数
   - `dropout` → 随机失活
   - `batch normalization` → 批归一化
   - `layer normalization` → 层归一化
   - `encoder` → 编码器
   - `decoder` → 解码器
   - `upsampling` → 上采样
   - `downsampling` → 下采样
   - `bounding box` → 边界框
   - `ground truth` → 真值标注
   - `feature pyramid` → 特征金字塔
   - `depthwise separable convolution` → 深度可分离卷积
   - `data augmentation` → 数据增强
   - `loss function` → 损失函数
   - `weight decay` → 权重衰减
4. 首次出现的重要概念可写成“中文（English）”，之后优先使用中文
5. 原文摘录始终保持英文，不翻译

### 4. 数学公式格式

论文中的数学表达式必须使用 LaTeX 语法，确保在 Obsidian / Typora 等 Markdown 渲染器中正确显示：

- **行内公式**：用 `$...$` 包裹，如 `$D(x)$`、`$p_g = p_{\text{data}}$`
- **独立公式**（核心目标函数、损失函数等重要公式）：用 `$$...$$` 包裹并独占一行
- **常见符号规范**：
  - 下标/上标：`$p_{\text{data}}$`、`$\theta_g$`、`$D^*(x)$`，不写成 `p_data`、`θ_g`、`D*(x)`
  - 数学算子：`$\min$`、`$\max$`、`$\log$`、`$\exp$`、`$\sum$`
  - 期望：`$\mathbb{E}$`，不写成纯文本 `E`
  - 分数：独立公式用 `$\frac{a}{b}$`，行内简单比值可用 `$a/b$`
- **判断标准**：文字中包含变量名、函数符号、下标上标或等式关系时，就应使用 LaTeX 包裹

### 5. 区块组装要求

#### 研究概述

- 不是机械复述 Abstract
- 用中文概括 4 个核心问题：**做什么问题、用了什么方法、创新点是什么、主要结果如何**
- 如果论文有明确 SOTA 提升、效率收益或泛化收益，写进概述

#### Figure 论证路径

- 直接使用阶段 2 的输出
- 确保结构与模板一致
- 如果阶段 2 标出了 `[图像未参考]` 或 `[作用待确认]`，保留这些提示，不要强行润色掉

#### 模型架构与训练

- 直接使用阶段 3 中 `## 模型架构与训练` 的内容
- 如果阶段 3 提供了 `### 推理设置（可选）`，保留在这一块中

#### Benchmark 与 Ablation

- 直接使用阶段 3 中 `## Benchmark 与 Ablation` 的内容
- 结果表格必须保持任务 / 数据集上下文

#### My Understandings

- **仅在交互模式下生成**
- 非交互模式：此区块完全不出现，不留空壳，不留占位符
- 交互模式内容来源：
  - **核心理解**：提炼用户在讨论中表达的理解
  - **纠正记录**：记录用户被纠正的认知（`❌ 原以为 → ✅ 实际上`）
  - **待解决的疑问**：用户提出但未完全解答的问题
- 只记录实际讨论过的内容，不补写未聊过的模块、Figure 或实验

### 6. 生成信息

在笔记末尾添加一行：

```text
> 由 paper-to-note skill 生成 | {当天日期 YYYY-MM-DD}
```

### 7. 质量控制

在组装过程中严格执行以下检查：

1. **删除 AI 语气** — 不允许出现“好的”“当然”“让我来解释”“接下来我们看看”等对话语气词
2. **不编造信息** — 不添加论文中不存在的模块、超参数、benchmark 数字或结论
3. **对话不是事实来源** — 交互讨论不能作为模型结构、训练配置、benchmark 结果或 ablation 结论的证据
4. **信息不足标注** — 信息不够时标注 `[信息不足，建议查阅原文]`，不要脑补
5. **格式一致性** — 所有表格、引用和标题层级保持统一
6. **中文检查** — 确认正文没有大段未翻译的英文学术概念；允许的英文仅限模型名、指标、数据集、模块专名和原文摘录
7. **无行内锚点** — 最终笔记中不允许出现任何证据编号、内部锚点或草稿标记

### 8. 文件名生成

格式：`{Venue缩写}--{年份}--{主要内容概括}.md`

规则：
- Venue 缩写中的空格用 `-` 替代（如 `CVPR`、`ICCV`、`TPAMI`）
- 主要内容用中文概括
- `--` 双连字符作分隔符
- 文件名不含空格
- 文件系统非法字符（`/ : * ? " < > |`）替换为 `-`

示例：`CVPR--2024--基于Transformer的实时目标检测.md`

## 输出格式

输出完整的 Markdown 笔记正文（不含代码块包裹），可直接写入 `.md` 文件。

## Notes

- 模板中没有的信息，不要强行补成“完整论文综述”
- 对 DL 论文来说，研究概述和 benchmark 结果通常是读者首先关心的部分，组装时要确保这两块清晰、紧凑、可回查
